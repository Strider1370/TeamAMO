# 07. 아이(아동) 도메인 — 같은 엔진, 다른 데이터 팩 (확장 설계)

> 목적: "부모 돌봄 내비게이터"와 **동일 엔진**으로 아동 사례를 커버할 수 있음을 설계로 못박는다.
> **MVP 범위 결정: 부모 도메인으로 좁힌다. 아이 도메인은 "확장 가능성"으로 발표에 노출**(엔진 일반화 증명). 데이터 빌드는 부모 우선.

## 1. 서비스 본질 (도메인 중립 재정의)
> 가족에게 **처음 겪는** 건강·돌봄·복지 문제가 생겼을 때, 상황을 말하면 AI가 의료·복지·교육·돌봄 정보를 **하나의 행동계획**으로 정리해주는 내비게이터.

부모/아이는 이 엔진의 **두 도메인 팩**일 뿐이다.

## 2. 공통 엔진 = 안 바뀌는 부분
| 공통 단계 | 부모 사례 | 아이 사례 |
|---|---|---|
| 사건 발생 | 입원·치매·퇴직 | 출산·희귀질환·발달지연 |
| 자연어 설명 | "아버지가 걷기 어려워요" | "아이가 말을 거의 안 해요" |
| AI 구조화 | 건강·돌봄·소득·주거 | 건강·발달·교육·돌봄 |
| 추가 질문 | 혼자 화장실 가능한가 | 어린이집에서 어떤 관찰을 했나 |
| 제도 연결 | 장기요양·기초연금 | 발달재활·특수교육 |
| 행동계획 | 퇴원 전·후 할 일 | 검사 전·진단 후 할 일 |
| 가족 공유 | 형제자매 역할분담 | 부모·조부모 역할분담 |

→ 엔진(자연어→슬롯추출→규칙매칭→의존성순서화→정직성→타임라인→공유)은 **그대로**. 바뀌는 것은 ① 상황 분류 키 ② `curated-benefits` 데이터 팩 ③ 규칙 테이블.

## 3. 아이 도메인 — 7개 상황 분류 (사용자 정의)
화면 진입: **"아이에게 어떤 일이 생겼나요?"**

| 키(가칭) | 상황 | 대표 제도(미검증 시드) |
|---|---|---|
| `출산직후` | 미숙아·NICU·선천성질환·난청/시각·심장/대사질환, 산모 동반치료 | 미숙아·선천성이상아 의료비지원, 산정특례, 재난적의료비, 병원 사회복지상담, 출산휴가·육아휴직, 신생아 난청검사/보청기 |
| `중증희귀` | 희귀유전질환·소아암·백혈병·근육병·대사질환·중증만성 | 산정특례, 희귀질환자 의료비지원, 재난적의료비, 장애등록, 보조기기, 병원학교(장기입원 아동 교육), 보호자 숙박·교통, 민간재단 |
| `발달지연` | 말 늦음·자폐 의심·발달장애·지적장애 | 영유아 발달평가, 발달재활서비스(바우처), 언어/놀이/감통치료, 장애등록, 특수교육대상자 선정, 발달장애인 부모상담 |
| `장애후유증` | 뇌성마비·뇌손상·사고후유증·감각/지체장애·소아뇌전증 | 장애등록, 장애아동수당, 발달재활, 장애인활동지원, 보조기기, 이동지원, 특수교육, 주거환경개선 |
| `교육생활` | 적응실패·분리불안·학교폭력·등교거부·학습장애·ADHD 의심·정서행동 | 학교상담, Wee센터, 정신건강복지센터, 아동발달검사, 학교폭력 신고·지원, 심리상담비, 돌봄교실·방과후 |
| `돌봄공백` | 한부모·사별·이혼·부모입원·산후우울·실직·교대근무·조부모양육 | 한부모가족지원, 아이돌봄서비스, 긴급돌봄, 어린이집 우선입소, 아동수당·부모급여, 긴급복지, 가족돌봄휴가 |
| `경제부담` | 장기치료비·발달치료비·보조기기·휴직 소득감소·교통비·주거불안 | 의료비지원, 재난적의료비, 아동수당, 장애아동수당, 발달재활 바우처, 교육비, 한부모지원, 긴급복지, 주거·에너지, 민간재단 |

> 아이 도메인의 특징(부모와 다른 점): **정보 파편화가 더 심함** — 병원(진단)·복지기관(치료지원)·교육청(특수교육)·주민센터(바우처)·어린이집(생활관찰)이 따로 논다. 그래서 "기관 역할 구분 + 무엇을 어디서 먼저"의 내비게이터 가치가 부모보다 오히려 더 크다. 또한 산출물에 **부모가 기록할 관찰·병원에 가져갈 기록·교사에게 물을 질문**처럼 "검사 전 준비"가 들어가야 함(독일 Pflegegrad 코칭 패턴과 동형).

## 4. 스키마 일반화 (재작업 없이 수용)
현 `BUILD-SPEC.md` 스키마는 도메인 중립이라 아이 제도도 그대로 수용한다. 추가/변경은 **단 1개 필드**:
- `domain`: `"parent" | "child"` — 부모 데이터엔 병합 단계에서 일괄 `"parent"` 주입(기존 10개 에이전트 산출물 무수정).
- `situations`: 아이용 7개 키는 위 §3 가칭 사용(부모 10키와 네임스페이스 분리).
- `target`: 아이 도메인은 `child`(아동 본인) / `caregiver`(부모) 2층 그대로 사용. 장애후유증처럼 **아이 지원 + 부모 지원 + 학교/어린이집 지원 3층**은 `target` + `situations` 조합으로 표현.

→ 결론: **엔진·스키마·UI 모두 재사용**. 아이 도메인 활성화 = 데이터 팩(`data/welfare/curated-child/*.json`) + 규칙 테이블만 추가하면 됨.

## 5. MVP 범위 결정 (왜 부모 우선)
- 5분 발표·4시간 빌드에서 **두 도메인을 다 깊게 = 둘 다 얕아짐**(심사 리스크: "넓지만 얕다"). → 부모 1도메인을 **깊게**(한 시나리오 데모 Wow) 가고, 아이는 **"같은 엔진, 데이터만 교체"** 로 확장 증명.
- 데모 옵션(스펙 단계에서 택1):
  - **(A) 확장 슬라이드만** — 위 공통구조표 + "아이 7분류" 한 장. 가장 안전. (권장 기본값)
  - **(B) 얇은 라이브 슬라이스** — 아이 1분류(예: `발달지연`) ~5제도만 실제 데이터로 채워, 데모에서 **도메인 토글**로 "같은 입력창, 데이터만 바꿈"을 라이브 시연. Wow 강하지만 +α 작업.
- 어느 쪽이든 **엔진/스키마는 지금 도메인 레디로 둔다**(domain 필드) — 그래야 확장 주장이 말뿐이 아니라 구조로 증명됨.

## 6. 자연어 예시 (아이 도메인, 데모용 가상)
- "아이가 태어나자마자 중환자실에 들어갔고 선천성 심장질환이라고 합니다. 병원비도 걱정이고 퇴원 후 뭘 해야 할지 모르겠어요." → `출산직후`+`경제부담`
- "아이가 30개월인데 말을 거의 안 하고 어린이집에서 자폐 검사를 받아보라고 했어요. 병원부터 가야 하는지 치료센터부터 알아봐야 하는지 모르겠습니다." → `발달지연`(다축: 의료·교육·복지·바우처 동시)
- "남편이 갑자기 입원해서 혼자 두 아이를 돌봐야 합니다. 직장도 다니는데 맡길 곳이 없어요." → `돌봄공백`

> 모든 아동 제도(미숙아 의료비·발달재활바우처·특수교육·아동수당·한부모지원 등)는 **빌드 시 출처 검증 필요**(현재 미검증 시드). 부모 도메인과 동일한 정직성 규칙 적용.
