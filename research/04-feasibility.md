# 트랙 D — 데이터·구현 현실성 (가족 복지·돌봄 내비게이터)

> 확인 시점: **2026-06-23** · 출처: 전부 레포 내 파일(경로 명시) + 일부 공식 제도(추가 확인 필요 표시).
> 결론 한 줄: **"검색"은 가진 스냅샷(~16,000건 복지/혜택)으로 덮인다. 그러나 "내비게이터의 킥"(평어→제도 언어 + 신청 순서 + 자격 판정)은 데이터만으로 안 되고, 대표 제도 10~15개를 손으로 룰화한 `curated-benefits`가 심장이다.** 4시간 빌드는 실시간 API 없이 정적 스냅샷+fixture로 충분히 데모 가능하다.

## 1. 주제 커버리지: 노인·치매·요양·돌봄을 가진 데이터로 얼마나 덮나

### 1-1. 데이터셋별 적합도 (이 주제 기준)

| 데이터셋 | 건수 | 이 주제 커버 | 매칭/체크리스트 유용 필드 | 한계 |
|---|---|---|---|---|
| **gov24-services** (보조금24) | 10,957 | ★★★ 본체 | `서비스명·서비스목적요약·지원대상·선정기준·신청방법·신청기한·접수기관·전화문의·**상세조회URL(gov.kr)**·지원내용·지원유형` | 자격이 **산문**(아래 §1-3). 노인/치매 전용 분류 코드는 없고 `서비스분야`(예: 보육·교육, 주거·자립) 수준 |
| **welfare-central** (중앙복지) | 452 | ★★☆ | `servNm·servDgst·jurMnofNm·**lifeArray**(영유아…노년)·**trgterIndvdlArray**(저소득·장애인·한부모 등)·intrsThemaArray·**servDtlLink(복지로)**` | `aplyMtdNm`(신청방법) 자주 비어있음(샘플 3건 중 0건 채움). 자격 정밀값 없음 |
| **welfare-local** (지자체복지) | 4,569 | ★★☆ + 지역 | 위 + `**ctpvNm·sgguNm**`(지역 필터) | 샘플상 `lifeArray/trgterIndvdlArray` **전부 빈 값**, `servDgst`는 채워짐 → 지역 필터 + 명칭/요약 검색만 신뢰 |
| **facilities** (사회복지시설) | 38,440 | ★★★ "다음 행동" | `name·typeCode·address·sido·sigungu·phone` (노인시설 ~24k) | **좌표 없음** → 거리정렬은 주소 지오코딩 필요. `typeCode`로 노인요양/주야간보호 등 필터 가능 |

### 1-2. `lifeArray`/`trgterIndvdlArray`가 매칭에 쓸 만한가 — **YES (중앙복지 한정)**
- 중앙복지 샘플에서 `lifeArray`에 **`노년`**이 실제로 박혀 있다(법률구조·안부살핌·수도요금감면 3건 모두 `노년` 포함). 즉 **"노년" 생애주기 토큰으로 1차 필터**가 가능하다 → "노인 대상 복지 모집단"을 코드로 좁힐 수 있음.
- `trgterIndvdlArray`엔 `저소득·장애인·한부모·조손` 등 **대상자 유형 토큰**이 있어 "저소득 노인" 교차 필터에 유용. 단 **치매/요양 등급 같은 토큰은 없다**(스키마에 그 축이 없음).
- ⚠️ **지자체복지는 이 두 필드가 샘플상 전부 비어 있어** 토큰 필터 불가 → 지자체는 `servNm/servDgst` 텍스트 includes + 지역(`ctpvNm/sgguNm`)으로만.

### 1-3. 결정적 한계: 자격은 "산문"이라 자동 판정 불가
gov24 `선정기준` 실제 값(샘플)을 보면 — 근로·자녀장려금은 `"단독가구 2,200만 원 미만 / 홑벌이 3,200만 원 / 맞벌이 4,400만 원..."` 같은 **자연어 단락**이고, 월세자금보증은 `선정기준: null`이다. 즉:
- **기계가 "이 사용자가 적격" 여부를 자동 판정할 수 없다.** → reference §1.3 원칙대로 **"잠재 적격 후보"까지만 정직하게** 보여주고 **공식 URL을 인용**한다.
- 진짜 개인 맞춤 판정(보조금24 맞춤안내)은 행정정보 연계라 외부 API로 안 옴(data-sources.md §교훈). 데모에서 흉내내면 거짓말이 된다.

## 2. 노인장기요양보험·치매안심센터, 이 카탈로그에 실제로 있나?

샘플만으로는 단정 불가하나, **스키마·소관기관 구조로 추론**하면:

| 제도 | 카탈로그 수록 가능성 | 근거/추론 |
|---|---|---|
| **노인장기요양보험(등급 인정 신청)** | △ **명칭 행은 있을 수 있으나 신청 절차가 산문/부재** | 보조금24엔 보건복지부·국민건강보험공단 소관 서비스가 다수 들어감. 단 "등급 신청 → 방문조사 → 등급판정 → 급여 이용"의 **순서**는 어느 필드에도 구조화돼 있지 않다 |
| **치매안심센터(등록·상담·검진)** | △ 명칭/요약 수준 가능, 운영주체는 보건소 | gov24/복지 텍스트 검색엔 걸릴 수 있으나, **"가까운 센터"는 facilities에 치매안심센터 typeCode가 따로 없을 가능성** → 별도 보강 필요 |
| **기초연금·노인맞춤돌봄·노인일자리** | ○ 수록 가능성 높음 | 보건복지부 대표 노인복지사업이라 보조금24/중앙복지에 명칭·링크는 있을 것 |
| **치매 치료관리비 지원·진단검사 지원** | △ | 명칭은 가능, 자격(중위소득 120% 등)은 산문 |

> **결론: "이름·링크는 카탈로그가 덮지만, 자격·신청 순서·다음 행동은 안 덮인다."** 이 빈틈이 정확히 우리 킥(평어→제도 언어 + 신청 순서 체크리스트)이 들어갈 자리다. → **curated-benefits로 손 보강 필수.**

### 2-1. 보완책 — `curated-benefits` 룰 테이블 (대표 제도 10~15개 손으로)
`data/welfare/curated-benefits.example.json` 양식(INDEX 기준: `id·name·rules(eligibility·slots)·applyUrl`)을 따라 **돌봄 주제 대표 제도를 직접 룰화**한다. 권장 수록 목록:

1. 노인장기요양보험 — 장기요양인정 신청
2. 치매안심센터 등록·상담·검진
3. 치매 치료관리비 지원
4. 기초연금
5. 노인맞춤돌봄서비스
6. 가사·간병 방문지원(재가급여)
7. 노인일자리·사회활동 지원
8. 긴급복지지원(위기 시)
9. 의료비 본인부담상한제 / 재난적의료비 지원
10. 노인주거복지(요양시설·주야간보호) 안내
11. 가족돌봄휴직(자녀 측 — 근로기준법/남녀고용평등법)
12. 장사·사망 관련(필요 시)

각 항목 최소 필드(제안 스키마):
```json
{
  "id": "ltci-apply",
  "name": "노인장기요양보험 장기요양인정 신청",
  "triggers": ["치매", "거동불편", "요양", "퇴원", "65세", "노인성질병"],
  "eligibility": "만 65세 이상 또는 65세 미만 노인성 질병(치매·뇌혈관성 질환 등)",
  "decisive": true,
  "steps": [
    "국민건강보험공단에 장기요양인정 신청",
    "공단 직원 방문조사(심신상태 조사)",
    "의사소견서 제출",
    "등급판정위원회 등급 판정(1~5등급·인지지원)",
    "표준장기요양이용계획서 수령 → 급여 이용"
  ],
  "documents": ["신분증", "의사소견서", "(대리신청시) 위임 관련 서류"],
  "applyUrl": "https://www.longtermcare.or.kr",
  "phone": "1577-1000",
  "source": "국민건강보험공단 노인장기요양보험 (확인 필요: 발표 전 공식 URL 재확인)"
}
```
> `applyUrl`/절차는 **발표 전 실제 접속 확인**(reference §1.3 "죽은 링크 주의"). 위 절차·번호는 일반 공지 기준이며 **빌드 시 공식 페이지로 검증 표시**.

## 3. 결정적 규칙 테이블 설계 초안 (상황 키워드 → 제도 매핑)

reference §1.3·§06-0 원칙: **AI는 평어 해독·값 추출만, 매핑·판정은 코드의 결정적 룰로 고정.** 자유 분류 금지.

| # | 사용자 상황 키워드(평어) | → 매핑 제도(결정적) | 비고 |
|---|---|---|---|
| 1 | 치매 / 기억이 깜빡 / 인지저하 | 치매안심센터 등록·검진 **+** 장기요양인정 신청 **+** 치매 치료관리비 | 치매는 2~3개 동시 점등 |
| 2 | 거동 불편 / 혼자 못 움직임 / 와상 | 장기요양인정 신청 **+** 가사·간병 방문지원 | |
| 3 | 독거 / 혼자 사심 / 고립 | 노인맞춤돌봄서비스 **+** 안부살핌 우편서비스(중앙복지 WLF00006237) | 중앙복지 실데이터에 존재 |
| 4 | 저소득 / 생활이 어려움 / 수급 | 기초연금 **+** 긴급복지지원 **+** 의료비 본인부담상한제 | `trgterIndvdlArray=저소득` 교차 |
| 5 | 퇴원 / 병원에서 나옴 / 회복 중 | 재가급여(방문요양·간호) **+** 재난적의료비 지원 **+** 주야간보호 시설 안내 | 퇴원=요양 진입 트리거 |
| 6 | 요양원 / 시설 / 모실 곳 | 노인주거복지 안내 **+** facilities(노인요양시설 typeCode 필터) | "다음 행동=시설 안내" |
| 7 | 65세 이상 / 노인 | 기초연금 + (life=노년 토큰으로 복지 모집단 확장) | 연령은 결정적 슬롯 |
| 8 | 간병하느라 일 못 함(자녀 측) | 가족돌봄휴직 안내 | 신청자=자녀 관점 |
| 9 | 의료비 부담 / 병원비 큼 | 본인부담상한제 + 재난적의료비 + (해당 시)차상위 본인부담경감 | |
| 10 | (지역 언급: OO시/구) | welfare-local을 `ctpvNm/sgguNm`로 필터해 지역 제도 추가 | 지역 슬롯 |

**구현 형태:** `triggers` 키워드 → `benefitId[]` 매핑은 **순수 코드 함수**(LLM 아님). AI는 사용자의 평어에서 키워드/슬롯(연령·지역·소득징후·상황)만 strict 도구로 추출(§06 set_field 패턴). 매핑·정렬·중복제거는 코드. → "틀려도 엉뚱한 결과가 안 나간다"(§1.4 신뢰).

## 4. 데이터 → 화면 파이프라인 (무엇이 무엇을 채우나)

```
사용자 평어 입력
   │ (AI: 키워드·슬롯 추출 — strict 도구)
   ▼
[결정적 룰 테이블]  ──→  curated-benefits (대표 10~15, 자격·순서·서류 구조화) ★체크리스트 본체
   │ 키워드→benefitId               
   ├──→ gov24-services (10,957)  명칭/요약/지원대상 includes → "잠재 후보" + gov.kr URL
   ├──→ welfare-central (452)    lifeArray=노년 + trgter 토큰 필터 → 복지로 URL
   ├──→ welfare-local (4,569)    ctpvNm/sgguNm 지역 필터 + 텍스트 → 복지로 URL
   └──→ facilities (38,440)      typeCode 노인요양/주야간보호 + sido/sigungu → "가까운 시설"(전화)
   ▼
맞춤 체크리스트: [제도명] → [자격(잠재)] → [신청 순서] → [준비서류] → [공식 URL/전화]
```
- **체크리스트의 "순서·서류"는 curated에서만** 신뢰성 있게 나온다(스냅샷엔 순서가 구조화 안 됨).
- 스냅샷은 **"이런 것도 받을 수 있어요(잠재 후보)"** 넓은 그물 + 공식 링크 역할.

## 5. 4시간 현실성 — 정적 스냅샷+fixture로 데모 가능한가

### 5-1. 가능 여부: **YES, 실시간 API 없이 충분**
- 이 주제는 data-sources.md 분류상 **"자주 안 바뀌는 위치/시설/조건 = 정적"** 쪽. 실시간 경보/측정값 불필요.
- API route에서 `fs`로 `../data/snapshots/*.json` 읽어 **서버사이드 필터 후 작은 결과만 반환**(INDEX "앱에서 읽는 법"). 스냅샷이 수MB~16MB라 클라로 통째 내려보내면 안 됨 — 서버 필터 필수.
- curated-benefits는 작은 fixture(수KB)라 그대로 import 가능.

```ts
// web/app/api/match/route.ts (개념)
import fs from "node:fs";
import path from "node:path";
const p = path.join(process.cwd(), "../data/snapshots/welfare-central.json");
const all = JSON.parse(fs.readFileSync(p, "utf8"));
const hits = all.filter(r =>
  r.lifeArray?.includes("노년") &&
  (r.servNm + r.servDgst).includes(keyword)
).slice(0, 20);
```

### 5-2. 가장 위험한 의존성 (우선순위 순)
1. **🔴 전수 스냅샷이 아직 레포에 없다.** `data/snapshots/*.json`은 미적재 상태. `dump-*.mjs`는 `DATA_GO_KR_KEY` 필요 + 중앙/지자체복지는 **활용신청 승인 필요**(미승인 시 Forbidden). **빌드 첫 30분의 단일 최대 리스크.** → 완화: ① 키·승인을 **전날 미리** 발급(data-sources.md 경고). ② 막힌 컨테이너 환경이면 `data.go.kr`이 `403 host_not_allowed`로 차단 → **사람이 미리 받아 `data/`에 둔 파일** 사용. ③ 최악엔 **샘플 3건 + curated만으로도 핵심 데모 성립**(스냅샷은 "넓은 그물"이라 없어도 킥은 산다).
2. **🟠 curated-benefits 손작업 품질.** 10~15개 제도의 자격·순서·서류·URL을 정확히 채우고 **링크 생존 확인**. 이게 데모의 신뢰성 그 자체. → 1~2명이 빌드 초반 병렬로 채움.
3. **🟡 facilities "가까운 시설" 거리정렬 — 좌표 없음.** 주소 지오코딩 필요. → **데모는 거리정렬 포기, `sido/sigungu` 일치 + 전화번호 노출**로 축소(시간 절약). 카카오 지오코딩은 욕심.
4. **🟡 자격 오판 리스크.** 산문 자격을 "적격"이라 단정하면 거짓. → UI 카피를 **"잠재 후보 · 공식 페이지에서 최종 확인"**으로 고정(§1.3).

### 5-3. 핵심 경로를 더 짧게 만드는 법
- **스냅샷 검색을 1차 데모에서 빼도 된다.** 평어 입력 → 룰 매핑 → **curated 체크리스트** 출력만으로 "검색이 아니라 내비게이터" 킥이 완성된다. 스냅샷(넓은 그물)·facilities(시설)는 시간 남으면 붙이는 **2차 레이어**.
- **지역 필터·지도·지오코딩은 전부 후순위.** 지자체복지/시설 거리정렬은 발표 데모에 불필요.
- **음성 입력도 폴백 우선.** 타이핑 입력으로 시연하고 음성은 데모 안전(§B-8) 차원의 옵션.
- 권장 4시간 배분: **(0~0:45) 스냅샷 적재 + curated 12개 작성(병렬)** / (0:45~2:30) API route + 룰엔진 + 체크리스트 UI / (2:30~3:15) gov24/복지 검색 레이어 + 시설 / (3:15~4:00) 카피·링크 검증·데모 리허설.

## 6. 요약 — "있는 것으로 덮을 것 vs 손으로 보완할 것"

**가진 스냅샷으로 덮인다:**
- 노년 대상 복지/혜택의 **넓은 모집단**(gov24 10,957 + 중앙 452 + 지자체 4,569)에서 명칭·요약·지원대상 텍스트 검색 + 공식 URL 인용.
- 중앙복지 `lifeArray=노년` / `trgterIndvdlArray=저소득` **토큰 필터**로 노인·저소득 교차 모집단 구성.
- 노인요양시설 "다음 행동" 안내(facilities 38,440, 전화번호) — 단 거리정렬은 제외.

**데이터로 안 덮여 손으로 보완해야 한다(킥의 심장):**
- **장기요양 등급 신청·치매안심센터 등록의 "신청 순서·준비서류"** → 스냅샷 어느 필드에도 구조화 안 됨 → **curated-benefits 10~15개 손작업**.
- **자격 판정** → 산문이라 자동 불가 → 결정적 룰(연령·지역·소득징후 슬롯) + "잠재 후보" 정직 표기.
- **평어→제도 언어 매핑** → §3 키워드 룰 테이블(코드 고정, AI는 추출만).
- **전수 스냅샷 적재 자체** → 아직 레포에 없음, 빌드 선행 과제(키·승인 전날 준비).

**실시간 API: 불필요.** 정적 스냅샷(서버 fs 필터) + curated fixture로 4시간 데모 충분히 가능. 단일 최대 리스크는 **스냅샷 미적재 + data.go.kr 승인/차단**이며, 완화책은 "전날 미리 받아 `data/`에 박제 + 최악엔 curated만으로 핵심 데모".
